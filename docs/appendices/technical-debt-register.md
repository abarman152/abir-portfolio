# Technical Debt Register

Findings from the 2026-07-03 full repository audit ([full report](./audit-report.md)). Each item lists severity, evidence, and recommended action. This register is the canonical list referenced by other docs — update it here, not in scattered inline notes, when debt is resolved or new debt is found.

| # | Item | Severity | Evidence | Recommended action |
|---|---|---|---|---|
| 1 | Docs claimed Cloudinary integration; none exists | Medium (docs accuracy) | No `cloudinary` package, no SDK usage anywhere in `backend/` or `frontend/`; admin image fields are plain URL inputs | **Fixed in this audit** — docs now describe it as a convention, not an integration. Decide: build real Cloudinary integration, or remove `CLOUDINARY_URL` from `.env.example` entirely. |
| 2 | Docs claimed a `{ data, error }` response envelope | Low (docs accuracy) | `frontend/AGENTS.md` §2.3 describes this envelope; actual routes return raw objects/arrays on success, `{ error: "message" }` on failure | Either adopt the envelope convention going forward, or fix `AGENTS.md` to match reality. Tracked here until decided. |
| 3 | `AGENTS.md` documents an `app/(public)/` route group that doesn't exist | Low (docs accuracy) | Routes are flat under `app/`, no parenthesized segment found | Fix `AGENTS.md`, or introduce the route group if there's a reason to group public routes. |
| 4 | Server-side validation is manual, not Zod, despite `AGENTS.md` §4.4 mandating Zod | Medium | `backend/src/routes/contact.ts` uses regex + length checks; `zod` is a frontend-only dependency, not present in `backend/package.json` | Either add Zod validation to backend routes, or update `AGENTS.md` to describe manual validation as the accepted convention. |
| 5 | `@prisma/client` present in frontend `package.json` | Medium (layer-bleed risk) | Contradicts `AGENTS.md` §5.2 "never import Prisma into the frontend" | Confirm it's unused, then remove from `frontend/package.json`. |
| 6 | Dead/unused dependencies | Low | `@react-three/fiber`, `@react-three/drei`, `three` (zero imports found), `next-themes` (custom `ThemeProvider.tsx` used instead), `axios` (native `fetch` used instead), backend `express-validator` and `multer` (no route uses them) | Remove from `package.json` after confirming no dynamic/conditional usage was missed. |
| 7 | Port default inconsistency | Low | `backend/.env.example` documents `PORT=5001`; `backend/src/index.ts` defaults to `5000`; most frontend pages fall back to `http://localhost:5001/api` but `frontend/src/lib/api.ts` falls back to port `5000` | Standardize on one port everywhere; always set `PORT`/`NEXT_PUBLIC_API_URL` explicitly to avoid relying on defaults. |
| 8 | Repeated API base-URL boilerplate | Low | `const API = process.env.NEXT_PUBLIC_API_URL \|\| 'http://localhost:5001/api'` duplicated verbatim in 13+ files instead of importing `lib/api.ts`'s base URL | Refactor call sites to use the shared `api.ts` client consistently. |
| 9 | No custom hooks layer | Low–Medium | Repeated stateful CRUD/toast/save-load patterns copy-pasted across every admin page (`frontend/src/app/admin/*/page.tsx`) instead of a shared hook | Extract `useAdminResource`/`useToast` hooks — see [`hooks/README.md`](../hooks/README.md). |
| 10 | Inline-style vs Tailwind inconsistency | Low | Admin pages (e.g. `admin/notifications/page.tsx`) use inline `style={{ }}` objects almost exclusively; public pages use Tailwind + CSS variables | Contradicts `AGENTS.md` §14 ("no inline styles unless unavoidable"). Migrate admin UI to Tailwind for consistency, or formally adopt inline styles as the admin-panel convention. |
| 11 | Homepage double-fetch | Low–Medium (performance) | `app/page.tsx` fetches server-side, then `HomePageClient.tsx` re-fetches the same data client-side via `clientFetch()` | Pass server-fetched data through as the initial client state instead of re-fetching. |
| 12 | No rate limiting on `/api/auth/login` | **Medium–High (security)** | `express-rate-limit` is applied only to `POST /api/contact`; login has no throttle/lockout | Add rate limiting (and optionally exponential backoff / account lockout) to the login route — brute-force risk. |
| 13 | Admin route protection is client-side only | Medium (security) | No `middleware.ts`; `admin/layout.tsx` only sets metadata; the actual guard is a `useEffect` in `AdminShell.tsx` checking `localStorage` | Consider adding a `middleware.ts` edge check (e.g., cookie-based) for defense in depth, especially since data itself remains JWT-gated regardless. |
| 14 | JWT stored in `localStorage`, not an httpOnly cookie | Medium (security) | Confirmed via `AdminShell.tsx`/`admin/notifications/page.tsx` reading `localStorage.getItem('admin_token')` | XSS-exposed token storage. Migrating to httpOnly cookies would require CSRF protection in exchange — a real architecture tradeoff, not a quick fix. Tracked in [`architecture/future-architecture.md`](../architecture/future-architecture.md). |
| 15 | Production `start` script pushes schema + reseeds on every boot | Medium (deployment risk) | `backend/package.json` `start`: `npx prisma db push && node dist/seed.js && node dist/index.js` | Split schema sync out of the boot path; use `prisma migrate deploy` in production per the deployment guide, and make `start` just run the compiled server. |
| 16 | `frontend/README.md` is unedited `create-next-app` boilerplate | Low | Confirmed — doesn't mention the project at all | **Should be fixed as part of this documentation project** (it's misleading to anyone opening `frontend/` directly). |
| 17 | Default admin credentials documented in plaintext in root `README.md` and `docs/development/setup-and-workflow.md` | Low (expected for a seed script, but worth flagging) | `admin@abirbarman.dev` / `Admin@123` | Already carries a "change immediately" warning; consider generating a random password at seed time instead and printing it once, to remove the temptation to skip rotation. |
| 18 | Stray `pr.txt` at repo root | Low (hygiene) | A saved PR description for the Resend migration, sitting at repo root instead of being deleted post-merge or moved into `CHANGELOG.md` history | **Resolved** — content preserved in [`releases/2026-notification-system-migration.md`](../releases/2026-notification-system-migration.md) and summarized in `CHANGELOG.md`; `pr.txt` (already gitignored, never tracked) has been deleted from disk. |
| 19 | Untracked `Archive/` directory at repo root | Low (hygiene) | Legacy branding/logo assets from before the rebrand, not part of the build, currently untracked | Added to `.gitignore` in this audit so it can't be accidentally committed. Recommend the user delete it locally once confirmed unneeded. |
| 20 | **Live database credential pasted into `backend/.env`** | **High (security — see [`security/secrets-management.md`](../security/secrets-management.md))** | A Supabase Postgres connection string with a plaintext password was found appended to `backend/.env`, apparently from a pasted terminal transcript | File is gitignored (never reached the remote repo), but **rotate the Supabase database password regardless**, since local files can leak via other channels. Flagged directly to the user; not modified by this audit. |
| 21 | Duplicate project-card / icon-set / Markdown-wrapper logic | Low–Medium | Near-identical `ProjectCard`-shaped rendering and icon-selection logic independently implemented in the home section, the about page, and the projects listing, found while writing component docs in Phase 6–8 | Extract a shared `ProjectCard` component (and a shared icon-lookup helper) under `components/ui/` or `components/shared/` and update all three call sites — see [`../components/README.md`](../components/README.md) for where each duplicate currently lives. |
| 22 | `Impact.tsx` is built but never rendered anywhere | Low (dead code) | `components/sections/Impact.tsx` exists and functionally overlaps with `Stats.tsx`, but no page imports it | Either wire it into a page if it's meant to replace/complement `Stats`, or delete it. Spawned as a separate follow-up task during the Phase 6–8 documentation pass; not resolved by this audit. |
